## 一、评分

打分维度：**技术准确性**（结论是否经得起源码核验）、**洞察深度**（是否抓到根因而非表象）、**可执行性**（优先级与落地路径是否清晰）、**覆盖完整性**。各 10 分。

| 报告 | 技术准确性 | 洞察深度 | 可执行性 | 覆盖完整性 | 总分 | 定位 |
|---|---|---|---|---|---|---|
| **claude-report-v1** | 9.5 | 9.0 | 9.0 | 7.5 | **8.8** | 工程收口派：最准、最聚焦 |
| **deepseek-report-v1** | 8.5 | 8.5 | 8.5 | 9.5 | **8.8** | 全景派：覆盖最广、根因抓得准 |
| **codex-report-v1** | 8.0 | 9.0 | 7.5 | 8.0 | **8.1** | 系统派：方法论独到、落地偏抽象 |

### 1.1 claude-report-v1（8.8）

**优点**——几乎每条结论都带行号且经核验属实：

- 2.1「卡住恢复只在 deep 生效」✅ 核实：`getCoastalSafeHeading` 仅在 `strategyId === 'deep'` 分支被调用（`autoNavigation.ts:1020-1021`），局部 A\* 绕行同样有 `=== 'deep'` 门控（`:840`）。常规策略停滞后只有 `useAlternateAxis` 单轴脱困（`:829-831`）。这是**全篇最高价值且最被低估的发现**，另两份报告都没明确点出"非 deep 策略缺自救"。
- 2.2「真实启动是无预算同步 A\*」✅ 核实：`createAutoNavigationPath` 默认 `maxSearchedNodes = Number.POSITIVE_INFINITY`（`:1065`），`startAutoNavigation` 不传预算（`actionsWorld.ts:63`），`findSeaPath` 同步阻塞主线程。
- 2.3 预览/实际口径不一致 ✅ 核实：预览 `createAutoNavigationPaths` 默认 `useCoastPenalty = false`（`:1536`），实际 `createAutoNavigationPath` 默认 `true`（`:1066`）。且指出 deep 已复用预览路径（`WorldMap.tsx:634`）而常规策略丢弃重算——这是别家没注意到的"重复计算"细节。
- 第 5 节测试缺口分析最扎实：明确核对了"仿真碰撞口径 = 世界态碰撞口径，当前一致无 mismatch"，直接**证伪了 codex 报告对碰撞模型分歧的隐含担忧**。

**扣分**：功能性扩展（自动进港、NPC、补给）几乎未覆盖——但这是刻意取舍，它选择把深度押在"已存在的工程债"上。

### 1.2 deepseek-report-v1（8.8）

**优点**——覆盖最全（架构/算法/功能/性能/鲁棒/UI/测试/文档八维），且抓住了**唯一的"根因型杠杆"**：

- 4.1 + 4.2「预计算离岸距离场」是三份报告里最好的单点洞察。它把"海岸惩罚 O(radius²) 环扫"这个底层瓶颈，与"预览被迫关惩罚 → 预览/实际不一致"这个上层 UX 问题**用一条根因串起来**，并给出具体数据（Lisbon→Mombasa 34ms→18.7s，来自 `AUTO_NAVIGATION.md`）和落地方案（2160×1080 Uint8Array ≈ 2.3MB，或降采样 145KB）。claude/codex 都只说"预览实际要统一"，但没指出"统一的前提是先消灭环扫成本"。
- 工作量估算（按天）和 2.1 分段混合精度寻路的方案最具体。

**扣分**：

- 部分高分项偏"功能蓝图"而非"当前缺陷"：NPC 舰队寻路、多目标贸易路线是新功能，体量大、非现状痛点，混在改进报告里拉高了噪声。
- 4.4 自己也承认"当前实现已做了该优化，提升有限"——这类项不该占篇幅。
- 对 2.1 这个真正的鲁棒性短板着墨不足。

### 1.3 codex-report-v1（8.1）

**优点**——方法论视角最独到，给了另两家完全没有的三个系统级武器：

- **P0 批量航线审计工具**（all-ports headless 健康检查）：这是从"逐条 case 补丁"升级到"地图级回归"的关键基建。论点"自动导航失败是一类海峡/群岛/港口入口问题，手写 case 必漏"成立，且能把失败收敛成 Jest fixture。**最有前瞻性的一条**。
- **保存路线版本校验**（`routeVersion`/`mapAssetVersion`/`algorithmVersion`）：唯一注意到"持久化的 path 在地图/算法/港口坐标变更后会失效穿陆"的报告。小成本、真实风险。
- 前瞻式 steering（每帧候选 heading 短距碰撞仿真打分）与 claude 2.1 殊途同归，但更进一步。

**扣分**：

- 技术核验最弱。P0「统一规划/移动碰撞口径」把 `isWorldSea()` 与 `map.collisionAt()` 描述为"很接近但不是同一抽象"，暗示存在分歧风险；但 claude 报告已逐行核对二者口径一致（tile≥50、2×2 占用），**当前并无 mismatch**。抽象统一有长期价值，但 codex 把它列为 P0 缺乏现状证据支撑。
- 多数建议停在"应该抽出 xxx 模块/应该结构化"，缺行号、缺成本估算、缺落地顺序细节。
- Web Worker 化深度搜索：方向对，但深搜已用 `setTimeout` 分块且非启动路径的主要冻结源（真正冻结源是 claude 2.2 指出的常规策略同步启动，codex 未提）。

## 二、三报告的共识与分歧

**强共识（三家都点名，可信度最高）**：
1. **预览路线与实际路线口径不一致**（claude 2.3 / deepseek 4.2 / codex P0-3）——已核实属实，UI 已挂免责声明（`WorldMap.tsx`），是确定要修的项。
2. `autoNavigation.ts` 过大、两套 A\* / 两套海岸惩罚重复（claude 3.1 / deepseek 1.1-1.3 / codex P1）——已核实 `findSeaPath` 与 `createChunkedSeaSearch` 各写一套。
3. 风向/洋流未进代价函数（三家均提）。

**关键分歧**：
- **碰撞口径**：codex 担心规划/移动分歧并列 P0；claude 核实当前一致。→ 采信 claude，降级为"重构时顺带统一的护栏"，非独立 P0。
- **最大瓶颈是谁**：deepseek 认为是海岸惩罚环扫成本；claude 认为是"卡住无法重规划"的鲁棒性。→ 二者层次不同，都对（见下）。

**各自独占的高价值点（互补，应全部纳入）**：
- claude 独占：**非 deep 策略缺通用卡住恢复**（最被低估）。
- deepseek 独占：**离岸距离场预计算**（最佳根因杠杆）。
- codex 独占：**批量航线审计工具** + **保存路线版本校验**。

## 三、最关键的改进（按"必须先做"排序）

收敛三份报告后，下面 5 项是真正的高杠杆。前两项是**地基**——不先建，后面的改动都在盲飞或带病加码。

### 🥇 P0-A：建立批量航线审计工具（codex 独占洞察）

**为什么排第一**：所有人都在讨论"该改什么"，但**没有人能证明改完没把别的航线改坏**。`AUTO_NAVIGATION.md` 的 bug 史反复出现"修好一条、破坏另一条"。在动任何寻路逻辑前，必须先有 all-ports × 代表港的 headless 健康检查：验证每个 waypoint 是海、相邻 waypoint 段不穿陆、抽样路线跑真实 `getAutoNavigationHeading()` 仿真不持续停滞，输出失败路线/段/坐标/策略/节点数/debug reason。

它同时是后续每一项的安全网与验收标准。**先建基线，再动刀。**
成本：中（2–3 天）。

### 🥈 P0-B：离岸距离场预计算（deepseek 根因杠杆）

**为什么排第二**：这是一块石头打下去能松动三处的根因。对 world tilemap 跑一次多源 BFS，存每个 tile 到最近陆地的距离（`Uint8Array`，全分辨率 ~2.3MB 或降采样 145KB），规划期 O(1) 查表替代 O(radius²) 环扫。它直接：
- 消除海岸惩罚性能瓶颈（claude 3.2 / deepseek 4.1）；
- 让**预览可以开海岸惩罚**，从而 P0-C 的"预览=实际"成为可能（deepseek 4.2）；
- 给每帧 `getReachedDistance`/开阔海面判定提速（claude 3.3）；
- 为风向代价改造（距离/航速）打地基（claude 2.4）。

成本：中（2–3 天）。

### 🥉 P0-C：统一预览与实际规划口径（三家共识）

依赖 P0-B 落地后顺势完成：预览直接用与真实启动相同参数（含海岸惩罚），点"自动导航"复用已算好的预览路径（deep 已这么做，常规策略照搬即可，`WorldMap.tsx:634`），去掉免责声明、省一次重算。
成本：低（叠加在 P0-B 上）。

### P1-A：非 deep 策略的通用"卡住恢复"（claude 独占洞察）

**这是当前最大的真实鲁棒性短板**。把"连续停滞 → 从当前位置对剩余目标做有界 A\* 重规划"和 `getCoastalSafeHeading` 从 deep 专属提升为所有策略共享（解除 `:840`、`:1020` 的 deep 门控）。直接消除"常规策略贴岸振荡到玩家手动取消"这一类投诉。必须配 P0-A 的卡死回归用例验收。
成本：中。

### P1-B：真实启动改为有界/分块（claude 2.2）

给常规策略真实启动设合理 `maxSearchedNodes` 上限并优雅降级，或复用 deep 的分块异步框架，消除低端机主线程冻结。
成本：中。

**随后再做**：合并两套 A\* / 海岸惩罚（三家共识，重构时统一碰撞口径作护栏）、风向代价、策略自动升级、路径平滑、保存路线版本校验（codex）。功能性扩展（自动进港、NPC、多目标、补给预估）价值真实但属于地基稳固后的迭代，不应抢占当前窗口。

## 四、一句话结论

三份报告分别贡献了不可替代的一块：**codex 给了"如何安全地改"（审计工具）**，**deepseek 给了"先撬哪块根因"（离岸距离场）**，**claude 给了"现状最痛的真实缺陷"（非 deep 策略卡死无救）**。最优路径不是三选一，而是按 **审计基建 → 距离场根因 → 预览统一 → 通用卡住恢复 → 有界启动** 的顺序串起来——前两步是地基，第三步顺势收 UX 债，后两步消除当前最高频的"卡住/卡顿"投诉。算法升级（双向 A\*、混合精度、风向 DP）应在地基稳固后再投入，收益才稳。
