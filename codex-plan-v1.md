# 自动导航报告评估与关键改进计划 v1

日期：2026-05-30

评估对象：

- `claude-report-v1.md`
- `deepseek-report-v1.md`
- `codex-report-v1.md`

## 结论先行

三份报告都抓住了自动导航的主要矛盾，但侧重点不同：

- `claude-report-v1.md` 最适合作为近期工程修复清单，问题定位具体，优先级比较稳。
- `deepseek-report-v1.md` 对性能瓶颈和玩家体验覆盖最广，但把一些体验功能排得过高，容易偏离当前最关键的可靠性问题。
- `codex-report-v1.md` 最强调系统化验证和碰撞一致性，方向正确，但低估了海岸惩罚性能对“预览与实际不一致”的根因作用。

我认为当前最关键的改进不是先加更多功能，也不是马上重写算法，而是建立一条“路线可信度闭环”：

1. 批量审计所有关键航线，得到可复现失败基线。
2. 统一规划、跟随、仿真测试、审计使用的船体通行判定。
3. 让 F4 预览路线、启动路线和实际跟随验证尽量使用同一条已验证路线。
4. 用离岸距离图替代规划期海岸环扫，解决开启海岸惩罚后的性能瓶颈。

这四步完成后，后续才适合投入风向代价、自动进港、NPC 航线、混合精度寻路等扩展。

## 当前代码事实

以下事实决定了新的优先级：

- `package.json` 没有 `audit:navigation` 脚本，当前仍缺少地图级批量航线健康检查。
- `createAutoNavigationPaths()` 默认 `useCoastPenalty = false`，F4 预览是快速近似路线。
- `startAutoNavigation()` 在没有传入 `plannedPath` 时调用 `createAutoNavigationPath()`，默认 `useCoastPenalty = true` 且 `maxSearchedNodes = Infinity`，常规路线启动仍可能同步阻塞主线程。
- deep 搜索已经用 `setTimeout` 分块，但常规 `findSeaPath()` 是同步 A*。
- `getAutoNavigationHeading()` 里局部 A* 绕行、近岸切轴、线段护栏主要集中在 `deep` 策略；常规策略遇到贴岸卡住时恢复能力弱。
- `isWorldSea()` 和 `map.collisionAt()` 都按 2x2 船体检查世界 tile，但它们是两套实现；真实移动还经过 `calculateDestination()` 的滑动规则，规划线段可通行不等于真实跟随可通行。
- 自动导航路径会随存档保存，但当前没有路线版本、地图版本或加载后重新校验机制。

## 最关键改进：路线可信度闭环

目标：任何显示给玩家、启动执行、保存恢复的自动导航路线，都必须能被同一套通行规则验证；失败能被批量审计发现，并能直接沉淀成回归测试。

这个目标比单独优化某条航线更关键，因为当前自动导航已经有不少特殊场景补丁。继续靠单条路线修复，会反复出现“修好 A 地，B 地又坏”的问题。

### P0.1 新增批量航线审计

新增脚本：

```bash
npm run audit:navigation
```

第一版不需要完美，但必须能批量输出失败基线：

- 遍历所有港口到所有港口，或先覆盖每个港口到代表性远洋、近岸、海峡目标。
- 对 `balanced`、`offshore`、`detailed` 至少运行规划层检查。
- 验证每个 waypoint 是否可停船。
- 验证相邻 waypoint segment 是否穿陆。
- 抽样运行真实跟随仿真：`getAutoNavigationHeading()` + `calculateDestination()`。
- 输出失败路线、策略、失败 waypoint、失败 segment、坐标、debug reason、搜索节点数。

完成标准：

- 有稳定 npm 脚本。
- 审计输出可排序、可复制到测试。
- 先提交当前失败数量作为基线，不要求第一步就清零。

### P0.2 抽出统一世界通行模块

新增或整理成类似模块：

```text
src/game/world/worldCollision.ts
```

建议提供：

- `isShipPositionNavigable(position)`：当前位置 2x2 船体是否可停。
- `canMoveShip(from, heading, speed)`：按真实移动规则模拟一步后是否前进。
- `moveShipForNavigation(from, heading, speed)`：审计和测试复用的移动结果。
- `isSegmentNavigable(from, to, options)`：路径段是否能被真实规则跟随。
- `getOffshoreDistance(position)`：后续由离岸距离图提供。

这一步不是为了抽象好看，而是为了让 A*、F4 预览、真实跟随、Jest 仿真、审计脚本不要各自维护一套近似规则。

完成标准：

- `isWorldSea()`、测试里的 `collisionAt()`、审计脚本不再各写一套 2x2 tile 判断。
- 现有自动导航测试仍通过。
- 新增至少一个测试证明规划判定和真实移动判定共用同一底层规则。

### P0.3 统一预览路线与启动路线

当前 F4 预览默认关闭海岸惩罚，启动时又重新按安全规则规划。这会让玩家看到的彩色路线和实际执行路线不一致。

建议改为：

- F4 预览产出的路线就是启动时执行的路线。
- 点击“自动导航”时优先传入选中的 `previewPaths[selectedStrategyId]`，避免丢弃预览重算。
- 如果启动前需要更严格验证，验证失败后必须立刻刷新 F4 上的实际路线，而不是默默换线。
- deep 路线已经复用 `deepRoutePath`，常规路线应采用同一原则。

完成标准：

- 常规策略启动时不再无条件同步重算一条不同路线。
- UI 文案不再依赖“实际航线可能略有差异”来解释系统不一致。
- 预览路线、启动路线和保存路线能通过 P0.1 审计规则。

### P0.4 用离岸距离图替代海岸惩罚环扫

`getCoastPenalty()` 当前对每个格子做半径 16 的环形扫描。即使有 per-search cache，长途路线首次计算仍重，导致预览为了响应速度默认关闭海岸惩罚。

建议预计算世界地图的离岸距离图：

- 对世界 tilemap 做多源 BFS 或距离变换。
- 用 `Uint8Array` 保存每个 tile 到最近陆地的大致距离，距离上限可截断到 255。
- 规划期只做 O(1) 查表，把海岸惩罚变成距离函数。
- deep 搜索和常规搜索共用同一距离数据，只通过参数调整惩罚半径和权重。

完成标准：

- 开启海岸惩罚后，常见远洋预览仍保持可交互。
- `getCoastPenalty()` 和 `getDeepGridCoastPenalty()` 不再各自环扫。
- 预览可以使用与实际启动一致的海岸安全口径。

## P1 改进

P0 完成后，再做以下内容才更稳：

1. 常规策略也使用近岸安全 heading 和有限重规划。
   - 当前 deep 策略在停滞后有局部 A* 和近岸切轴，常规策略主要依赖单轴脱困。
   - 推广时必须有路径长度上限和审计保护，避免历史上“局部绕行无限增长”的问题复发。

2. 拆分 `autoNavigation.ts`，但先保持行为不变。
   - 建议拆成 grid、pathfinding、deepSearch、follower、debug、strategy。
   - 拆分应服务后续修改和测试，不应和算法重写混在一个变更里。

3. 路径平滑。
   - 只在 `isSegmentNavigable()` 通过时删除中间 waypoint。
   - 远洋段可大胆平滑，近岸和窄水道保守处理。

4. 路线质量指标。
   - 显示总长度、绕行系数、预计天数、补给风险、近岸危险段。
   - 风向/洋流可先进入估算和展示，再进入 A* 代价。

5. 保存路线增加版本与加载校验。
   - 保存 `routeVersion`、`mapAssetVersion` 或 tilemap hash、`algorithmVersion`。
   - 加载旧路线时先校验，失败则从当前位置重算或取消并提示。

## 暂缓项

以下项目有价值，但不应排在路线可信度闭环之前：

- 自动进港：体验收益高，成本低，但不能解决路线不可信问题。
- NPC 舰队寻路：属于世界行为扩展，依赖稳定的导航核心。
- 多目标贸易路线：功能量大，依赖路线保存、补给估算和到港行为。
- 双向 A*：算法收益不确定，且要处理横向环绕、海岸惩罚和启发式一致性。
- Web Worker：可以改善主线程体验，但如果没有先减少海岸惩罚成本和统一验证，只会增加调试复杂度。
- 风向/洋流进入 A* 代价：很适合提升路线质量，但应在通行可靠后做。

## 建议实施顺序

1. 新增 `npm run audit:navigation`，建立失败基线。
2. 抽出统一世界通行模块，并让测试/审计复用。
3. 让常规 F4 预览路线直接用于启动，消除预览和实际路线分叉。
4. 实现离岸距离图，替换海岸惩罚环扫。
5. 用审计结果挑选最高频失败海域，补回归测试。
6. 推广常规策略近岸恢复和有限重规划。
7. 再做 `autoNavigation.ts` 行为保持式拆分。

## 成功标准

短期成功标准：

- 能一条命令得到自动导航健康报告。
- 常规路线启动不再默默换成另一条未展示路线。
- 规划、跟随、测试和审计共用同一套船体通行判定。
- 审计中的失败路线可以快速转成 Jest 回归测试。

中期成功标准：

- 开启海岸惩罚的预览仍能保持流畅。
- 常规策略贴岸卡住数量明显下降。
- 存档恢复后的自动导航路线会被校验，而不是盲目信任旧 path。

最终判断：

自动导航下一阶段的核心任务是“可信”，不是“更多”。先把批量验证、通行一致性、预览/启动一致性和海岸惩罚性能打通，后续所有算法和体验改进才有可靠落点。
