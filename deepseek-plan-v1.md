## 二、最关键改进项识别

以下通过三个报告的交叉验证，按"用户可感知影响 × 实现可行性 × 风险降低"三角评估，筛选出最关键的 5 项改进。

### 2.1 🔴 关键项一：通用卡死恢复机制（来源：Claude 独占洞察，Codex 隐含支撑）

**问题本质**：`getAutoNavigationHeading` 中，停滞恢复逻辑（局部 A\* 绕行、`getCoastalSafeHeading` 近岸切轴）仅对 `deep` 策略开放。`balanced`/`detailed`/`offshore` 三种常规策略一旦在近岸/海峡楔住，唯一的自救手段是单轴脱困——这在复杂海岸线前几乎无效。结果是船贴着海岸来回振荡，直到玩家手动取消。

**为什么这是 #1**：
- 直接影响玩家体验（"自动导航卡住了"），且无法自行恢复
- 修复成本中等，不需要新算法，主要是把已有逻辑从 deep-only 提升为策略无关
- 不修复的话，其他任何算法改进都无法惠及常规策略用户

**改造路径**：
1. 将"连续停滞超过阈值 → 从当前位置对剩余目标做有界 A\* 重规划"提升为所有策略共享的通用恢复逻辑
2. 让常规策略在停滞 N 次后也调用 `getCoastalSafeHeading`（当前被错误限制在 deep 分支）
3. 重规划使用当前策略的网格精度 + 有限节点预算（如 5000 节点），避免 deep 的开销
4. 若局部重规划也失败，升级到下一级策略（见 2.5 自动策略升级）

---

### 2.2 🔴 关键项二：预览路线即实际路线（来源：三报告全部 P0/5 星）

**问题本质**：F4 预览（`useCoastPenalty = false`、节点预算有界）和实际启动（`useCoastPenalty = true`、节点预算无界）是两套不同的规划口径。玩家看到的蓝色/绿色/紫色预览线与出发后的黄色实际航线可能不同。界面甚至需要额外的免责文字。

**为什么这是 #2**：
- 直接造成玩家认知混乱——"我选的是这条线，为什么走的是另一条？"
- 预览通过不代表实际能通过，增加失败风险
- 三个报告无一例外将其列为最高优先级，共识最强

**改造路径**：
1. **短期**（不依赖性能优化）：非 deep 策略在点"自动导航"时，直接复用预览阶段已算好的路径（deep 已经这么做了），而非丢弃重算
2. **中期**（依赖 2.5 的预计算距离场）：预览也开启海岸惩罚（查表 O(1) 代替环扫），使预览本身就是真实路线的忠实近似
3. 去掉 F4 底部的免责说明

---

### 2.3 🔴 关键项三：统一"规划可通行"与"真实移动可通行"判定（来源：Codex 独占洞察）

**问题本质**：规划用 `isWorldSea()` 判断 2×2 船体占用，真实移动通过 `map.collisionAt()` + `calculateDestination()` 做碰撞滑动。两套逻辑虽然目前数据口径一致（都基于 tile ≥ 50），但不是同一抽象，且移动的斜向"先横后纵"轨迹不等于规划的网格直线。后续若给世界态加特殊地形/水深/沉船，极易出现"规划认为能走，船实际撞岸"的分歧。

**为什么这是 #3**：
- 这是许多"卡住"类 bug 的**根本原因**——不是算法找不到路，而是找的路实际走不了
- 统一后，A\*、路径审计、局部避障、测试仿真、地图 UI 都复用同一组规则
- Claude 在测试覆盖分析中也暗示了此风险（"规划口径与真实口径若将来分叉"）

**改造路径**：
1. 新建 `src/game/world/worldCollision.ts`（或扩展现有 `map.ts`），抽象出：
   - `isShipPositionNavigable(position)` — 船体当前位置是否可停
   - `canShipMove(from, heading, speed)` — 模拟真实移动规则后是否可通行
   - `isSegmentNavigable(from, to, options)` — 线段是否可被真实跟随（即当前的 `isSegmentSea` + 船体宽度模拟）
2. A\* 的 `isSea`/`isSegmentSea` 改为调用上述统一模块
3. 测试的 `collisionAt` 也改为调用同一模块
4. 后续增加特殊地形时只需改一处

---

### 2.4 🟡 关键项四：批量航线审计工具（来源：Codex 独占洞察）

**问题本质**：当前测试靠手写案例覆盖高价值路线，但自动导航的失败不是单点 bug，而是某类海峡/群岛/港口入口/粗网格连通性的系统性缺陷。只靠人工案例会遗漏大量路线，且修好一条可能破坏另一条。

**为什么这是 #4**：
- 它不是用户直接感知的功能，但它是**保证其他改进不引入回归的前提**
- 优先级被 Claude 和 DeepSeek 完全忽略，是最大的集体盲区
- 有了审计基线，后续任何算法改动都可以跑 `npm run audit:navigation` 验证不退化

**改造路径**：
1. 新建 headless 审计脚本 `scripts/audit-navigation.ts`
2. 覆盖：所有港口 → 所有港口（或每个港口到若干代表港口）
3. 每条路线验证：waypoint 可通行、segment 无穿陆、抽样仿真不持续停滞
4. 输出 JSON 报告：失败路线、失败 segment、坐标、策略、节点数、停滞位置

---

### 2.5 🟡 关键项五：海岸距离场预计算 + 策略自动升级（来源：三报告综合）

**问题本质（两个互补问题）**：
- **海岸距离场**：`getCoastPenalty` 半径 16 的逐格环扫是性能瓶颈（单格最坏 ~1089 次 `isSea`，开海岸惩罚后预览耗时从 34ms 涨到 18.7s），直接导致预览被迫关海岸惩罚，进而造成预览/实际不一致。
- **策略自动升级**：若 `balanced` 失败，玩家需手动切 `offshore`/`detailed`/`deep`。普通玩家不知道这些策略的区别，只会觉得"自动导航坏了"。

**为什么捆绑为一项**：距离场是基础设施——解决后不仅预览可开海岸惩罚（助攻 2.2），还能加速每帧的海面判定（`isOpenSeaForDiagonalHeading` 169 格扫海），降低跟随层 CPU 开销。策略自动升级则是利用已有的策略梯度，让玩家不需要理解内部机制。

**改造路径**：
1. 对整张 world tilemap（2160×1080）做一次多源 BFS，生成 `Uint8Array` 离岸距离图（~2.3MB，与 tilemap 同量级）；或降采样到 4×4 分辨率（540×270，~145KB）
2. 所有需要海岸距离的地方改为 O(1) 查表
3. 策略失败时自动按 `balanced → offshore → detailed → deep` 升级，全程状态提示

```
Phase 1 (1-2 周)：止血 — 消除用户体验最大痛点
├── 2.1 通用卡死恢复机制 ← 所有策略共享停滞重规划

├── 2.5a 海岸距离场预计算 ← 预览可开海岸惩罚 + 每帧查表加速
├── 2.4 批量航线审计工具 ← 建立回归基线
└── 2.5b 策略自动升级 ← balanced→offshore→detailed→deep

Phase 3 (2-4 周)：重构 — 降低维护成本
├── 合并两套 A\* / 海岸惩罚 ← 根治"只改一边"bug
├── 拆分 autoNavigation.ts ← 为 Worker 迁移做准备
└── 路径平滑（string-pulling）← 利用已有 isSegmentSea

Phase 4 (远期)：能力升级
├── 风向/洋流代价 (Level 1: 跟随层选顺风方向)
├── 深度搜索迁 Web Worker
├── 前瞻式避障 ← 替换反应式停滞检测
├── 自动进港 + 补给风险评估
└── 路线版本校验
```

## 五、一句话总结

三份报告从不同视角审视了同一系统——Claude 最懂代码痛点（卡死恢复），DeepSeek 覆盖面最广（功能补全），Codex 工程视角最强（批量审计、统一碰撞）。**交叉验证后的最关键结论是：当前系统的瓶颈不在算法能力，而在于 (1) 常规策略缺少卡死恢复、(2) 规划与移动的通行判定分裂、(3) 预览与实际路线不一致。** 这三项是用户每天会遇到的问题。在此基础上，海岸距离场预计算作为基础设施投入，能同时解决预览/实际不一致、跟随层性能、以及为风向代价等远期改进铺路。批量审计工具则保证所有改动可验证、不回归。五管齐下，2-4 周内可将自动导航从"能用但常卡"提升到"可靠且可维护"。
