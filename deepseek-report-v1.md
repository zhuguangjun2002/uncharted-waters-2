## 一、代码架构与可维护性

### 1.1 `autoNavigation.ts` 过于庞大，建议拆分

**现状：** 1578 行的单文件包含了所有自动导航逻辑——A\* 寻路、二叉堆、海岸惩罚、航向计算、分块搜索、深度路由、预览路径生成。

**问题：**
- 文件职责过多，新人难以快速定位
- `findSeaPath()` (349–563) 和 `createChunkedSeaSearch()` (1137–1392) 有大量重复的邻居展开逻辑
- `getAutoNavigationHeading()` (805–1059) 混合了航点推进、停滞检测、绕路、线段通海护栏等多项职责

**建议：**
```
src/game/world/
  autoNavigation.ts          # 保留：策略定义、公共入口 (createAutoNavigationPath / createAutoNavigationPaths)
  aStar.ts                   # 新建：findSeaPath、二叉堆、邻居展开
  chunkedSearch.ts           # 新建：createChunkedSeaSearch、findDeepRoutePath
  heading.ts                 # 新建：getAutoNavigationHeading、getDirectionToPosition、getCoastalSafeHeading
  navigationUtils.ts         # 新建：isWorldSea、isSegmentSea、getCoastPenalty、canMoveInDirection
```

**优先级：** ⭐⭐⭐ 中等 — 不影响功能，但会显著提升可维护性

### 1.2 `createChunkedSeaSearch` 与 `findSeaPath` 存在大量重复代码

**现状：** 两个函数各自实现了几乎相同的 A\* 邻居展开循环、二叉堆操作、`isGridSea` 包装、`cameFrom`/`gScore`/`closedSet` 管理。

**差异点：**
- `createChunkedSeaSearch` 多了 `returnClosestOnExhaust`、分块执行、`onProgress` 回调
- `findSeaPath` 多了 `useSegmentClearance`、端点豁免
- 海岸惩罚实现略有不同（`getGridCoastPenalty` vs `getDeepGridCoastPenalty`）

**建议：** 将核心 A\* 逻辑抽取为共享函数，两个入口通过参数控制不同行为。

**优先级：** ⭐⭐⭐ 中等

### 1.3 海岸惩罚有两套实现

**现状：**
- `getCoastPenalty()` (280–310)：16 半径、4 档惩罚 (20/8/3/1)，供 `findSeaPath` 使用
- `getDeepGridCoastPenalty()` (1197–1234)：3 半径、3 档惩罚 (3/2/1)，供 `createChunkedSeaSearch` 使用

**问题：** 底层逻辑相同（环形扫描→找最近陆地→按距离分级），因参数不同而重复实现。

**建议：** 统一为一个参数化的 `getCoastPenalty(position, isSea, { maxRadius, penaltyTable })`。

**优先级：** ⭐⭐ 低

## 二、算法与寻路

### 2.1 分段混合精度寻路（AUTO_NAVIGATION.md 已提及）

**现状：** 所有策略都是"整条路线同一种网格精度"。`balanced` 尝试 8×8 全程 → 失败则 4×4 全程，不存在"海峡用 4×4、远洋用 12×12"的混合路线。

**问题：**
- 马六甲海峡、直布罗陀海峡等窄水道需要细网格才能表示，但远洋段用细网格会过于贴岸且探索节点过多
- 当前用 `useSegmentClearance` 护栏阻止粗网格"跳过"地峡，但无法解决粗网格在海峡处的"盲区"

**建议方案：**
1. 先用 12×12 粗网格规划远洋段骨架
2. 识别骨架中经过窄海峡/近岸的段落（通过 `getCoastPenalty` 判断）
3. 仅对这些段落用 4×4 细网格重新规划
4. 拼接混合精度路径

**难点：** 如何准确识别"需要细网格的段落"而不漏判或误判；拼接点如何保证连续性。

**优先级：** ⭐⭐⭐⭐ 高 — 这是从"能用"到"好用"的关键一步，也是文档中明确提到的后续方向

### 2.2 双向 A\*（AUTO_NAVIGATION.md 已提及）

**现状：** 所有 A\* 搜索都是单向的（从起点向终点扩展）。

**问题：** 长途航线（如 Lisbon → Macao）的单向 A\* 探索节点数为 O(b^d)。双向 A\* 理论上可降至 O(b^(d/2))。

**建议：**
- 在 `findSeaPath` 中增加 `useBidirectional` 选项
- 从起点和终点同时向中间扩展，两个 open set 相遇时拼接路径
- 优先用于超远航线的 4×4 粗网格搜索（该阶段节点数最多）

**潜在问题：**
- 世界地图 x 方向环绕使"相遇条件"更复杂
- 两个方向的启发式必须一致（可逆），当前启发式（octile 距离）满足条件
- 海岸惩罚破坏了启发式的一致性（使得双向拼接不能保证最优），需要在双向模式下关闭海岸惩罚

**优先级：** ⭐⭐⭐ 中等 — 能在特定场景显著加速，但需要处理好 x 环绕和启发式一致性

### 2.3 风向/洋流感知航线优化（AUTO_NAVIGATION.md 已提及）

**现状：** 航线规划完全不考虑风向和洋流。A\* 的代价函数只包含距离和海岸惩罚。

**问题：** 游戏有完整的风/流数据（`src/data/assets/windsCurrent.wasm`、`src/game/world/windCurrent.ts`），实际航行速度受风影响很大。逆风航线虽然"最短"但实际耗时可能更长。

**建议方案（渐进式）：**
- **Level 1：** 在 `getAutoNavigationHeading()` 中，当有多个可选航向时（开阔海面斜向 vs 单轴），优先选择顺风方向
- **Level 2：** 在 A\* 代价函数中加入风向因子：逆风边的代价 ×1.5，顺风边 ×0.8
- **Level 3：** 考虑季节性风带变化（如贸易风带随季节南北移动），为长途航线规划最佳出发时间

**难点：** 风/流数据是 8×8 区域粒度且分季节，需要高效查询；风向与航向的"顺逆"关系需要精确定义。

**优先级：** ⭐⭐⭐ 中等 — Level 1 成本很低但体验提升明显

### 2.4 超远航线路径后处理

**现状：** 超远航线的 tile 级路径直接使用，不做后处理。由于关闭了海岸惩罚且航点间隔只有 1 格，waypoint 可能贴在离岸 1–2 格的位置。

**问题：** 贴岸 waypoint 在实际跟随中容易被船体碰撞挡住，虽然局部 A\* 绕行能脱困，但反复触发绕行会增加航行抖动和耗时。

**建议：** 找到路径后做一次后处理扫描：
1. 对每个 waypoint 计算离岸距离（通过 `getCoastPenalty`）
2. 对于离岸 < 某个阈值（如 2 格）的 waypoint，向外推至附近最远的开阔水域
3. 确保推出后相邻 waypoint 之间仍然线段通海

**优先级：** ⭐⭐ 低

## 三、功能完整性

### 3.1 自动进港

**现状：** 自动导航到达目标港附近后停止，玩家必须手动按 `E` 进港。

**问题：** 
- 长途航行可能持续数分钟，玩家可能在 AFK，到港后船停在港外浪费时间
- 不符合原作"一键到达"的体验

**建议：**
1. 到达目标港附近（进入 `REACHED_TARGET_DISTANCE`）后自动调用 `dock()`
2. 在 F4 面板增加一个"自动进港"开关（默认开启）
3. 到达时显示提示、播放音效

**难点：** 需要确保船确实在可进港位置（`positionAdjacentToPort` 判定），否则自动进港会失败。

**优先级：** ⭐⭐⭐⭐ 高 — 高频需求，显著改善体验

### 3.2 NPC 舰队寻路

**现状：** `worldNpc.ts` 中的 NPC 舰队使用 `randomDirection()` 随机移动，不向任何目标寻路。

**问题：** NPC 船队在海面上随机徘徊，不符合原作中 NPC 舰队在港口之间航行的行为。

**建议方案（渐进式）：**
- **Level 1：** NPC 在预设港口对之间巡逻，使用 `createAutoNavigationPath('balanced')` 
- **Level 2：** NPC 舰队有自己的贸易路线逻辑（在港口买卖货物）
- **Level 3：** 敌对 NPC（海盗）主动追踪玩家舰队

**难点：**
- NPC 寻路不能使用分块/异步模式（需要同步返回 heading），需要限制节点预算
- 多个 NPC 同时寻路可能有性能问题 → 可以错帧执行或使用更粗的网格
- 需要考虑 NPC 舰队可见性（是否在玩家视野内决定更新频率）

**优先级：** ⭐⭐⭐ 中等 — 提升世界生动性，但不是 MVP 必需的

### 3.3 多目标贸易路线

**现状：** 一次只能设置一个目标港。

**问题：** 原作的核心玩法之一是规划贸易路线（如 Lisbon → Mediterranean ports → back），自动导航不支持。

**建议：**
- 状态扩展：`autoNavigation.waypoints: PortId[]`
- F4 UI 增加"添加经停港"按钮
- 到达一个港口后自动设置下一个目标
- 每个经停港可选择"自动进港"或"仅经过"

**优先级：** ⭐⭐ 低 — 量大但属于锦上添花

### 3.4 自动补给

**现状：** 补给（食物、水）需要手动在港口的 Harbor 购买。长途航行可能在中途耗尽。

**问题：** 自动导航不会检查补给是否足够到达目标港。

**建议：**
- 启动自动导航前估算所需天数（航线长度 / 平均船速）
- 对比当前补给量 → 不足时提示"补给可能不足，是否继续？"
- 可选：在 F4 面板显示预估到达时间和所需补给
- **不做**自动购买补给（会引入复杂的港口交互逻辑）

**优先级：** ⭐⭐⭐ 中等 — 防止玩家 AFK 回来后发现船队饿死

## 四、性能优化

### 4.1 海岸惩罚计算是性能瓶颈

**现状：** `getCoastPenalty()` 对每个未缓存的格子做环形扫描，最大半径 16 → 最坏情况扫描约 4×16² ≈ 1024 次 `isSea` 调用。每个 `isSea` 又检查 4 个 tile 偏移。

**实测数据（来自 AUTO_NAVIGATION.md）：**
- 开海岸惩罚后 Lisbon → Mombasa 预览从 34ms 涨到 18.7s
- 因此 F4 预览被迫关闭海岸惩罚

**建议：**
- **短期：** 增加预计算的离岸距离图（offshore distance map）。可以预先对 world tilemap 做一次 BFS 标注每个 tile 到最近陆地的最短距离，存为 `Uint8Array`（2160×1080 ≈ 2.3MB，与 tilemap 同大小）。查询时直接查表 O(1)。
- **长期：** 如果预计算图太大，可以降采样到 4×4 分辨率（540×270 = 145KB），读表时用线性插值。

**优先级：** ⭐⭐⭐⭐⭐ 非常高 — 解决了预览与实际路径不一致的核心矛盾

### 4.2 F4 预览与实际航线的路径不一致

**现状：** F4 预览（`createAutoNavigationPaths`）默认 `useCoastPenalty = false` 以保持速度，实际启动时（`startAutoNavigation`）才用 `useCoastPenalty = true` 重新规划。这导致预览路径可能与实际航线不同，F4 底部需要额外文字说明。

**问题：** 玩家看到蓝色路线很好，但实际航行时黄色路线偏离了预览，造成困惑。

**建议：** 一旦 4.1 的预计算离岸距离图落地，预览就可以开海岸惩罚（查表而非扫描），预览路径与实际路径完全一致。

**优先级：** ⭐⭐⭐⭐⭐ 非常高 — 依赖 4.1

### 4.3 `isWorldSea` 可加 tile 级缓存

**现状：** `isWorldSea` 每次调用检查 4 个 tile 偏移。`getCoastPenalty` 环形扫描中大量重复查询相邻坐标。

**建议：** 使用基于坐标键的 LRU 缓存（如 `Map` 限制最大 10,000 条目），因为环形扫描有很强的空间局部性。

**优先级：** ⭐⭐ 低 — 配合 4.1 的预计算图后可能不再需要

### 4.4 深度搜索的正向/反向泛滥可并行

**现状：** `findDeepRoutePath()` (1415–1529) 先跑正向泛滥(`forward`)，拿到结果后再跑反向泛滥(`backward`)，是顺序的。

**问题：** 两个方向的泛滥互不依赖，完全可以并行。

**建议：** 使用 `Promise.all([forward.promise, backward.promise])` 同时发起两个方向的粗网格搜索。

**注意事项：**
- 两个搜索共享同一个 `onProgress` 回调，需要区分来源（或直接累加）
- 如果正向已经到达目标（普通港口场景），不需要跑反向 → 所以应该先 `await forward.promise`，仅当 `!forwardResult.reachedTarget` 时才启动反向

**优先级：** ⭐⭐ 低 — 当前实现已经做了这个优化（先检查 forward 是否到达），性能提升有限

## 五、边缘情况与鲁棒性

### 5.1 双端窄水道港口

**现状（AUTO_NAVIGATION.md 已记录）：** 如果起点和目标**都是**被窄水道封住的港口（各自在小连通块中，主海域是第三方），`findDeepRoutePath()` 会优雅返回空。当前实现假设至少有一端在主海域。

**问题场景举例：** 理论上的极端情况——例如 Changan（1 tile 宽水道）到另一个同样窄水道的港口。现实中极少见，但从鲁棒性角度应考虑。

**建议：**
1. 当两端泛滥的节点数都**不大**（说明两端都不是主海域），但两端的连通块**不重合**（说明是三个独立连通块），需要额外跑一次从主海域任意点（如 Lisbon 邻海）的泛滥来确认主海域
2. 然后两端各做 tile 级接驳到各自的水道口，长途段走主海域
3. 考虑到实际航线中这种场景极少，可以暂时标记为"已知限制"，不做代码修改

**优先级：** ⭐ 极低

### 5.2 局部 A\* 绕行使路径无限增长

**现状（AUTO_NAVIGATION.md 已记录）：** 曾尝试将局部 A\* 绕行推广到粗策略（balanced/offshore），导致路径在长航线上无界增长 → OOM。

**现状缓解：** 当前绕行只对 `deep` 策略启用，且每次绕行目标按物理距离选取（≥ 100px），不会无限增长。

**建议：** 增加路径长度上限保护——如果 `newPath.length > 原路径长度 × 2`，放弃绕行、改用 `skipCount` 跳点。

**优先级：** ⭐⭐ 低 — 当前已有缓解措施，作为安全网补充

### 5.3 小地图极端高纬度港口显示

**现状：** Nome（x=2062, y=156，近北极）在 F4 小图上可以正常显示，但北极圈以上的位置（y < 50）在小图上只有 1–2 像素高。

**问题：** 如果未来有更北的港口（或玩家传送到极地），F4 小图的港口标记可能与地图边缘重合。

**建议：** 暂无实际需求，记录为已知限制。

**优先级：** ⭐ 极低

## 六、UI/UX

### 6.1 进度指示不直观

**现状：** 进度按 `已通过 waypoint 数 / 总 waypoint 数` 计算。

**问题：**
- deep 策略航点密集（每 1–4 px 一个），总航点数可达 500+。船在开阔海域能快速跳过航点（64px 到达半径），进度条可能瞬间跳到 30%–40%
- 但在近岸水道缓慢推进时，进度条长时间不动
- 用户无法感知"离实际到达还有多远"

**建议：**
- 改用物理距离进度：`当前位置到目标港的距离 / 起点到目标港的距离`
- 额外显示估算时间：`剩余距离 / 当前平均速度`
- 在 F4 面板上用小地图方式直观显示"船当前位置 → 目标港"的相对位置

**优先级：** ⭐⭐⭐ 中等

### 6.2 补给风险评估

**现状：** 无。如果玩家在 Lisbon 只剩 10 天食物但目标港是 Macao（需要数月），自动导航不会警告。

**建议（关联 3.4）：**
- 启动自动导航时计算预估天数
- 对比当前补给 → 不足时弹出警告
- F4 面板增加一行补给状态（如"粮食: 45 天 / 预估需要: 60 天 ⚠️"）

**优先级：** ⭐⭐⭐ 中等

### 6.3 F4 世界地图可增加信息密度

**现状：** F4 小图显示真实颜色的缩略地图、港口位置、航线、当前位置。不显示地名、不显示风/流、没有"迷雾"（发现状态）。

**建议（渐进式）：**
- **Level 1：** 鼠标悬停港口时显示港口名（而非仅列表中显示）
- **Level 2：** 在小图上叠加风向/洋流箭头（类似原作风格）
- **Level 3：** 实现"发现状态"——未访问过的海域在图中显示为暗色

**优先级：** ⭐⭐ 低 — Level 1 成本很低

### 6.4 深度搜索期间缺乏取消反馈

**现状：** 深度搜索可中途取消（`DeepRouteHandle.abort()`），搜索过程中用脉冲进度条表示正在计算。

**问题：** 如果搜索时间较长（如 5–10 秒），用户可能不确定系统是否卡死。

**建议：** 
- 显示实时节点探索速率（如"每秒探索 50,000 节点"）
- 如果 5 秒内进度条不动（`onProgress` 未触发），显示"搜索中，请耐心等待..."
- 增加 Esc 快捷键取消深度搜索

**优先级：** ⭐⭐ 低

## 七、测试覆盖

### 7.1 仿真测试速度较慢

**现状：** 部分仿真测试运行时间较长：
- `Lisbon to Mombasa balanced actual-nav`：~10s
- `Lisbon to Mombasa offshore actual-nav`：~5s

**问题：** 开发时每次跑全量测试都要等 15s+。

**建议：**
- 将仿真测试拆到单独的 `autoNavigation.sim.test.ts` 文件
- 单元测试（`autoNavigation.test.ts`）只跑纯逻辑（findSeaPath、getDirectionToPosition、getAutoNavigationHeading），不加载 `worldTilemap.wasm`
- 仿真测试可标记为 `slow`，CI 中运行但本地开发可选跳过（`jest --testPathPattern='(?<!sim\.test)'`）

**优先级：** ⭐⭐⭐ 中等

### 7.2 缺少 F4 WorldMap 组件的渲染测试

**现状：** `WorldMap.tsx` 有 1081 行，包含复杂的 Canvas 绘制逻辑（`drawBaseMap`、`drawPath`、`drawAutoNavigation`、地图接缝处理），但没有对应的单元测试。

**建议：**
